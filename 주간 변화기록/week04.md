# 4주차: 창의프로그래밍입문설계 개발 로그
**프로젝트명:** Survival Blue Ball (가제) - 충돌 엔진 고도화 파트
**작성일:** 2026-03-24

## 1. 개요
기존의 단순한 원형 피격 판정을 넘어, 게임 에셋(칼)의 회전과 형태에 대응하는 정밀한 충돌 감지 시스템을 구축했습니다. 특히 사각형의 회전을 고려한 OBB(Oriented Bounding Box)와 SAT(분리축 이론)를 도입하여 상용 게임 수준의 판정 로직을 구현하는 데 집중했습니다.

## 2. 주간 개발 및 변화 과정

### [1단계] 외부 에셋 통합 및 스프라이트 시스템 구축
- **기능:** `Sprites.py` 모듈을 연동하여 단순 도형이 아닌 실제 게임용 '칼(Sword)' 이미지를 렌더링함.
- **로직:** `pygame.transform.scale`을 통해 에셋 크기를 규격화하고, `blit`을 이용한 이미지 출력 시스템으로 전환함.

### [2단계] 동적 회전 로직 구현
- **기능:** 플레이어의 입력이나 마우스 방향이 아닌, 프레임당 일정 수치(`rotation_speed`)만큼 스스로 회전하는 자동 회전 무기 로직 구현.
- **수학:** `pygame.transform.rotate` 사용 시 발생하는 이미지 크기 변화 문제를 `get_rect(center=...)`를 통해 중심점을 고정하여 해결함.

### [3단계] 3종 충돌 방식(Circle, AABB, OBB) 동시 시각화
- **원형(Circle):** 파란색 가이드라인으로 표시. 두 중심점 사이의 거리를 계산하여 판정.
- **AABB:** 빨간색 사각형으로 표시. 회전과 관계없이 오브젝트를 감싸는 축 정렬 사각형 판정.
- **OBB:** 초록색 다각형으로 표시. 오브젝트의 회전 각도를 그대로 반영하여 가장 정밀하게 감싸는 판정 영역 구현.

### [4단계] SAT(분리축 이론) 기반의 정밀 충돌 엔진 도입
- **기능:** 회전된 두 직사각형이 겹치는지 판단하기 위해 SAT(Separating Axis Theorem) 알고리즘을 직접 코딩함.
- **수학:** 각 변의 법선 벡터를 투영축으로 설정하고, 모든 축에서 투영 영역이 겹치는지 검사하여 '진짜 충돌' 여부를 판별함.

### [5단계] 실시간 디버그 UI 및 피드백 시스템
- **UI:** 화면 왼쪽 상단에 각 충돌 방식의 결과(HIT / IDLE)를 텍스트로 실시간 출력.
- **피드백:** 충돌 방식에 따라 텍스트 색상을 가이드라인 색상과 일치시켜 가시성을 확보함.

## 3. 최종 결과 요약
단순히 "닿았다"는 느낌을 주는 단계를 넘어, **수학적 모델(Circle, AABB, OBB)**이 실제 게임 내에서 어떻게 다르게 작동하는지 비교 분석할 수 있는 환경을 만들었습니다. 특히 OBB 구현을 통해 회전하는 근접 무기의 타격 판정을 정밀하게 제어할 수 있는 기술적 기반을 마련했습니다.

## 4. 질의응답 및 오류 해결 (Q&A)

### Q1. 이미지 회전 시 발생하는 중심점 이탈 및 떨림 현상
- **문제:** `rotate` 함수 사용 시 이미지의 전체 크기가 변하면서 좌측 상단(`topleft`) 좌표가 틀어져 칼이 제자리에서 떨리는 현상 발생.
- **해결:** 회전된 이미지의 새로운 `Rect`를 생성할 때, 기존의 `center` 좌표를 강제로 주입하는 `get_rect(center=center_move)` 방식을 사용하여 회전축을 고정함.

### Q2. 이미지 회전 방향과 OBB 박스 회전 방향의 불일치
- **문제:** 시각적인 칼 이미지는 시계 방향으로 도는데, 수학적으로 계산한 초록색 OBB 박스는 반시계 방향으로 돌아 판정이 어긋남.
- **해결:** `math.cos`와 `math.sin` 계산 시 각도 값에 마이너스(`-rad_angle`)를 적용하여 파이게임의 좌표계와 회전 방향을 물리적으로 동기화함.

### Q3. 파이썬 들여쓰기(IndentationError) 및 코드 구조 설계
- **문제:** 복잡해진 루프 안에서 특정 로직이 루프 밖으로 삐져나와 프로그램이 실행되지 않는 오류 발생.
- **해결:** 이벤트 처리, 상태 업데이트, 충돌 계산, 화면 그리기의 4단계 레이어로 코드를 구조화하고, 전체 선택 후 들여쓰기를 재정렬하여 파이썬의 구문 규칙을 준수함.

### Q4. OBB 충돌 연산 시의 성능 부하 고려
- **문제:** 매 프레임 다수의 법선 벡터를 계산하고 투영하는 SAT 방식이 저사양 환경에서 성능을 저하시킬 우려가 있음.
- **해결:** 가장 가벼운 `Circle` 판정을 먼저 수행(Broad-phase)하고, 원형 영역이 겹칠 때만 정밀한 `OBB` 판정을 수행(Narrow-phase)하는 2단계 검증 구조를 제안함.
